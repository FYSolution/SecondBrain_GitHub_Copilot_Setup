# Second Brain — Wiki Operating Schema

## Project: [Your Project Name]

## Purpose

Persistent, **team-shared** project memory. Analysis, summary, indexing, and long-term
knowledge retention for the entire development team.

This system is NOT about coding behavior (that's governed by Karpathy Guidelines separately).

Multiple LLM agents (one per developer) maintain a structured wiki that compounds knowledge
over time. The file structure is designed to **minimize merge conflicts** when multiple
developers commit wiki updates concurrently.

---

## Core Principles

1. **Knowledge compounds** — Every ingest, query, and code task makes the wiki richer.
   Query answers that produce novel synthesis are promoted to `wiki/analysis/` pages,
   so explorations accumulate alongside ingested sources. Nothing valuable stays
   trapped in ephemeral chat history.
2. **Maintenance cost ≈ zero** — The LLM handles all cross-referencing, summarizing,
   and bookkeeping. Humans curate sources, direct analysis, and ask good questions.
3. **Persistent artifact > ephemeral chat** — The wiki is the durable output. Chat
   sessions are transient; their value must be captured in wiki pages before the
   session ends.
4. **Synthesis over retrieval** — Unlike RAG, which re-derives answers from raw chunks
   on every query, this wiki pre-compiles knowledge. Cross-references are already
   built, contradictions already flagged, and the synthesis already reflects every
   source ingested.

---

## Team Collaboration Design Principles

1. **Per-user files for high-churn content** — logs, journals, and code-updates use
   `{username}` in the filename so concurrent commits never touch the same file.
2. **Shared pages use additive sections** — entity/concept pages are shared but edits
   are append-to-section (not rewrite), reducing conflict surface.
3. **Auto-generated index** — `wiki/index.md` is regenerated by script, never hand-edited.
4. **Atomic log entries** — each log entry is a separate file (not appended to one file).
5. **Last-writer-wins for synthesis pages** — `overview.md`, `lessons.md`, entity pages
   are collaborative; PRs resolve rare conflicts via standard merge review.

### Username Convention

Each developer's LLM uses a consistent username identifier. This is set in the
developer's local `.env` or derived from their git `user.name`:

```
SECOND_BRAIN_USER=fyang
```

If not configured, fall back to git: `git config user.name` → lowercase, no spaces
(e.g., "Frank Yang" → "frankyang").

## Architecture

| Layer   | Path        | Owner  | Rule                                                |
| ------- | ----------- | ------ | --------------------------------------------------- |
| Schema  | `SCHEMA.md` | Team   | Operating manual, co-evolved via PRs                |
| Raw     | `raw/`      | Human  | Immutable — LLM reads but NEVER modifies            |
| Wiki    | `wiki/`     | LLM(s) | LLM-owned — creates, updates, cross-references      |
| Scripts | `scripts/`  | Team   | Automation helpers (index gen, conflict resolution) |

### Raw Sources (`raw/`)

Human-curated, immutable once added. This is the source of truth.

| Folder              | Contents                                               |
| ------------------- | ------------------------------------------------------ |
| `raw/requirements/` | Business requirement documents (BR\*.md)               |
| `raw/design/`       | Architecture & design decision documents               |
| `raw/sessions/`     | Chat transcripts / session notes (dropped in by user)  |
| `raw/decisions/`    | Meeting notes, client feedback, scope changes          |
| `raw/code-updates/` | Code change reports (per session/commit)               |
| `raw/analysis/`     | Solution reports, security scans, performance analysis |
| `raw/architecture/` | Solution structure snapshots, service inventory        |

### Wiki (`wiki/`)

LLM-generated and LLM-maintained. Multiple LLMs can write concurrently.

| File/Folder            | Ownership  | Purpose                                         |
| ---------------------- | ---------- | ----------------------------------------------- |
| `wiki/index.md`        | Script-gen | Auto-generated catalog (DO NOT edit manually)   |
| `wiki/log/{user}/`     | Per-user   | Each user's append-only operation log           |
| `wiki/overview.md`     | Shared     | Project architecture synthesis, current state   |
| `wiki/lessons.md`      | Shared     | Accumulated corrections, project-specific rules |
| `wiki/entities/`       | Shared     | One page per service/component                  |
| `wiki/concepts/`       | Shared     | One page per cross-cutting pattern              |
| `wiki/sources/`        | Shared     | One summary page per ingested raw source        |
| `wiki/analysis/`       | Shared     | Promoted query answers, comparisons, syntheses  |
| `wiki/requirements/`   | Shared     | Synthesized BR status, groups, changelog        |
| `wiki/journal/{user}/` | Per-user   | Each user's daily session summaries             |

### Conflict-Free Zones (per-user, never conflicts)

- `wiki/log/{user}/YYYY-MM-DD.md` — user's daily log entries
- `wiki/journal/{user}/YYYY-MM-DD.md` — user's daily journal
- `raw/code-updates/{user}-YYYY-MM-DD.md` — user's code change reports

### Shared Zones (rare conflicts, resolved via PR review)

- `wiki/entities/*.md` — entity pages (use additive edits)
- `wiki/concepts/*.md` — concept pages (use additive edits)
- `wiki/sources/*.md` — source summary pages
- `wiki/analysis/*.md` — promoted query answers and syntheses
- `wiki/overview.md` — project overview
- `wiki/lessons.md` — shared lessons
- `wiki/requirements/status-matrix.md` — BR tracking

## Operations

### Session Start (Context Loading)

Before responding to the first task:

1. Read `wiki/index.md` (page catalog — find relevant pages)
2. Read own log: `wiki/log/{user}/` (latest file — recent personal context)
3. Scan other users' recent logs: `wiki/log/*/` (last entry each — team awareness)
4. Read `wiki/lessons.md` (project knowledge, mistakes to avoid)
5. Read relevant `wiki/entities/` page for the current task
6. **Auto-ingest un-ingested sources** — compare files in `raw/` (recursively, excluding
   `.gitkeep` and `raw/code-updates/`) against pages in `wiki/sources/`. If any raw
   file has no corresponding source page, **automatically ingest them all** using the
   Ingest operation (see below). After ingestion completes, notify the user:

   > "Auto-ingested X new raw source(s): [list with brief one-line summary each]."

   Do NOT wait for user confirmation — ingest immediately at session start.
   If ingestion fails for a specific file (e.g., binary, unreadable), skip it and
   report the failure in the notification. Then proceed with the session normally.

### Task Completion (Auto Wiki Update)

**After completing a code modification task, update the wiki as the FINAL step.
This is part of task completion, not a separate request.**

> ⚠️ **Scope:** This applies to ALL tasks that create, modify, or delete files —
> including changes to the wiki system itself (SCHEMA.md edits, script enhancements,
> structural refactors). There is no "meta-work" exception. If you touched files,
> you track it.

1. **CREATE/APPEND** `raw/code-updates/{user}-YYYY-MM-DD.md`:

   ```markdown
   ## [HH:MM] Task Description

   ### Files Modified

   - path/to/file.cs — reason for change

   ### Decisions

   - What was chosen and why

   ### DevOps

   - Task #ID (if mentioned)
   ```

2. **CREATE/APPEND** `wiki/journal/{user}/YYYY-MM-DD.md`:
   - Synthesized summary (concise bullets, not raw dump)
   - Include `**Author:** {user}` header

3. **UPDATE** relevant `wiki/entities/` or `wiki/concepts/` pages:
   - Service gained new capability → update entity page
   - Pattern changed → update concept page
   - Add `<!-- updated: {user} YYYY-MM-DD -->` comment at end of modified section

4. **UPDATE** `wiki/requirements/status-matrix.md` if a BR progressed

5. **APPEND** to `wiki/log/{user}/YYYY-MM-DD.md`:

   ```
   ## [HH:MM] action | description
   ```

6. **ALWAYS** run `scripts/generate-index.ps1` to regenerate `wiki/index.md`.
   This is unconditional — run it at the end of every task, not just when new pages are created.
   It is cheap to run and guarantees the index stays consistent.

### Ingest (triggered by: "ingest [path]")

When user adds a new raw source and asks to ingest:

1. Read the raw source completely
2. Summarize key points to user
3. **CREATE** `wiki/sources/{slug}.md` — a dedicated summary page for this source
   (see Source Summary Page Template below)
4. Identify which wiki pages need updating
5. Update/create wiki pages with cross-references using [[wikilinks]]
6. **Flag contradictions** — if new source contradicts existing wiki content, add a
   `> ⚠️ **Contradiction:** [new claim] vs [existing claim from [[page]]]` callout
   in the affected wiki page(s)
7. Append to `wiki/log/{user}/YYYY-MM-DD.md`: `## [HH:MM] ingest | source-name`
8. **ALWAYS** run `scripts/generate-index.ps1` to regenerate `wiki/index.md`

### Lint (triggered by: "lint wiki")

Health-check the wiki for:

- [ ] Raw sources not yet recorded in any user's log (un-ingested)
- [ ] Raw sources without a corresponding `wiki/sources/` page
- [ ] Wiki pages with stale information vs actual code
- [ ] Orphan pages (no inbound links from other pages)
- [ ] `requirements/status-matrix.md` out of sync with code
- [ ] Contradictions between entity pages (check `> ⚠️ **Contradiction:**` callouts)
- [ ] Unresolved contradictions older than 2 weeks
- [ ] Missing entity pages for services that exist in code
- [ ] lessons.md entries that are no longer relevant
- [ ] Conflicting edits between users on shared pages
- [ ] Pages missing YAML frontmatter
- [ ] Stale `sources` arrays in frontmatter (raw file deleted or renamed)
- [ ] Important concepts referenced in entity/concept pages but lacking their own dedicated page (scan body text for repeated bolded terms or [[wikilinks]] that resolve to nothing)
- [ ] Knowledge gaps where a targeted web search or additional raw source could fill missing details — suggest concrete queries or source types to the user

### Query (triggered by questions about the project)

1. Read `wiki/index.md` to find relevant pages
2. Read those pages
3. Synthesize answer with citations to raw sources
4. **Auto-update wiki** — if the answer revealed new knowledge about a service,
   pattern, or feature, update the relevant entity/concept page immediately.
   Do NOT wait for user to ask "update wiki".
5. **Promote by default** — any substantive query answer (comparison, analysis,
   synthesis, design exploration, root-cause investigation) is filed as a new page
   in `wiki/analysis/` (e.g., `wiki/analysis/comparison-auth-schemes.md`).
   Promotion is the default, not the exception. Skip promotion ONLY when the
   answer is trivially short (single fact lookup), fully duplicates an existing
   page, or the user explicitly says "don't file this". When in doubt, promote —
   a stub page is cheaper than a lost insight.

## Merge Conflict Resolution

### Prevention (by design)

| Content Type  | Strategy                          | Conflict Risk |
| ------------- | --------------------------------- | ------------- |
| Logs          | Per-user files                    | None          |
| Journals      | Per-user files                    | None          |
| Code-updates  | Per-user files                    | None          |
| Index         | Script-generated                  | None          |
| Entity pages  | Additive sections, comment stamps | Low           |
| Concept pages | Additive sections, comment stamps | Low           |
| Overview      | Single synthesizer per PR         | Low           |
| Status matrix | Row-level edits                   | Low           |
| Lessons       | Append-only entries               | Low           |

### When Conflicts Occur (shared pages)

1. Git will show standard merge conflict markers
2. **Both changes are likely valid** — merge both, don't discard
3. For entity/concept pages: keep both sections, deduplicate if redundant
4. For status-matrix: accept the more-progressed status
5. For lessons: keep all entries (append-only by design)

## Scaling

The index-based approach (LLM reads `wiki/index.md` to find relevant pages) works
well up to ~150 pages. Beyond that, add search tooling.

### Thresholds & Actions

| Wiki Size     | Strategy                                                                                  |
| ------------- | ----------------------------------------------------------------------------------------- |
| < 150 pages   | Index file is sufficient. No additional tooling needed.                                   |
| 150–500 pages | Add `scripts/search-wiki.ps1` — keyword search with ranked results. LLM shells out to it. |
| 500+ pages    | Add dedicated search tool (qmd, MCP search server, or similar).                           |

### Health Check

Run periodically to assess whether tooling upgrades are needed:

```powershell
$count = (Get-ChildItem Second_Brain/wiki -Recurse -Filter *.md).Count
Write-Host "Wiki page count: $count"
if ($count -gt 100) { Write-Host "⚠️ Approaching threshold — consider building search-wiki.ps1" }
```

### `scripts/search-wiki.ps1`

Keyword search over all wiki markdown files. Implemented — see `§ Scaling` for
usage notes and threshold context.

- Multi-word queries use AND logic
- Output is a ranked markdown table (file path, score, matching snippet)
- Supports `-Folder` flag to scope to `entities`, `concepts`, etc.
- Supports `-Top N` to limit results

```powershell
& ".\Second_Brain\scripts\search-wiki.ps1" "keyword1 keyword2" -Top 10 -Folder entities
```

### Planned: future search tooling

When page count crosses ~500, evaluate moving to a dedicated search tool
(qmd, MCP search server, or similar).

## Conventions

### General

- Every wiki page starts with a one-line summary (after frontmatter)
- Use [[wikilinks]] for cross-references between pages
- Keep pages focused — one service OR one concept per page
- Bullet points over prose
- Concise — no filler text
- Add `<!-- updated: {user} YYYY-MM-DD -->` comment when editing shared pages
- Use inline citations: `[↗ raw/path/to/source.md]` to trace claims back to sources

### YAML Frontmatter

All wiki pages (except index.md and log entries) SHOULD include frontmatter:

```yaml
---
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: [raw/requirements/BR01.md, raw/design/auth-design.md]
tags: [entity, service, notification]
---
```

- `created` — date page was first created
- `updated` — date of last substantive edit
- `sources` — list of raw source paths that informed this page
- `tags` — freeform tags for categorization and filtering

### Entity Page Template

```markdown
---
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: [raw/requirements/BR01.md]
tags: [entity, service]
---

# [Service Name]

One-line: what this service does.

## Purpose

The service does X for Y reason [↗ raw/requirements/BR01.md].

## Key Methods

- `MethodName(args)` — what it does [↗ raw/design/auth-design.md]

## Status Transitions (if applicable)

## Related BRs

## Recent Changes

<!-- updated: {user} YYYY-MM-DD -->

- [change description]

## Links

- [[related-concept]]
- [[related-entity]]
```

> **Cite every non-trivial claim** with `[↗ raw/path/to/source.md]` so any wiki
> statement is traceable back to the source that informed it. This is the same
> rule for concept pages, analysis pages, and source summary pages.

### Source Summary Page Template (`wiki/sources/`)

```markdown
---
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: [raw/path/to/original-source.md]
tags: [source, requirement|design|decision|analysis]
---

# [Source Title]

One-line: what this source is about.

## Key Takeaways

- Bullet points of the most important information
- Claims that update or extend existing knowledge

## Contradictions Found

- (if any) What this source says vs what the wiki previously stated

## Pages Updated

- [[entities/service-name]] — added X capability
- [[concepts/pattern-name]] — revised Y behavior

## Raw Source

- [↗ raw/path/to/original-source.md]
```

### Analysis Page Template (`wiki/analysis/`)

```markdown
---
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: [wiki pages and raw sources consulted]
tags: [analysis, comparison|synthesis|investigation]
---

# [Analysis Title]

One-line: what question this analysis answers.

## Context

Why this analysis was done (original query or trigger).

## Findings

## Conclusion

## Links

- [[related-pages]]
```

### Journal Entry Format (per-user)

```markdown
# YYYY-MM-DD

**Author:** {user}

## Tasks Completed

- [description]

## Code Changes

- [file]: [what and why]

## Decisions

- [choice]: [rationale]

## DevOps

- Created: #IDs
- Completed: #IDs

## Open Items

- [unresolved questions]
```

### Log Entry Format (per-user file)

```
## [HH:MM] action | description
```

Actions: `session`, `ingest`, `lint`, `query`, `promote` (query answer filed as analysis page)

### Lessons Format

```markdown
## [YYYY-MM-DD] Category — {user}

- What went wrong or was corrected
- The fix or correct approach
- Rule to follow going forward
```

## Scripts

### `scripts/generate-index.ps1`

Scans `wiki/entities/`, `wiki/concepts/`, `wiki/requirements/`, `wiki/journal/*/`
and regenerates `wiki/index.md`. Run after creating new pages.

### `scripts/merge-logs.ps1`

Combines all `wiki/log/*/YYYY-MM-DD.md` into a single chronological timeline
view for review. Restores the grep-able single-log property of the original
Karpathy LLM Wiki pattern (per-user files prevent merge conflicts; this view
recombines them).

- Output: `wiki/.merged-log.md` (read-only, not committed — add to `.gitignore`)
- Normalized format: `## [YYYY-MM-DD HH:MM] {user} | action | description`
- `-Tail N` flag prints only the last N entries to stdout (no file write)

```powershell
& ".\Second_Brain\scripts\merge-logs.ps1" -Tail 20
```

## Notes

This schema is intentionally focused on wiki/memory operations only.
Coding behavior, quality, and discipline are governed by Karpathy Guidelines
(`.github/instructions/karpathy-guidelines.instructions.md`) — a separate, always-active system.

The team-safe design trades slight verbosity (per-user folders) for **zero merge conflicts**
on high-churn files. Shared knowledge pages (entities, concepts) still benefit from
collaborative editing — conflicts there are rare and meaningful (two devs changed the
same service), resolved through normal PR review.
